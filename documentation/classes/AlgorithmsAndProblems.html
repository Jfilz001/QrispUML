Algorithms and Problems

Information about classes and methods

The 'Algorithms and Problems' diagram collects the high-level problem and benchmark classes that represent complete quantum algorithms. These classes act as entry points for users: they bundle a problem Hamiltonian or cost structure together with the corresponding quantum circuit ansatz or operator, an optional classical optimiser, and helper methods for running, training, and benchmarking. The diagram covers three algorithm families – VQE, QAOA/QIRO, and DCQO – plus the quantum backtracking framework.

VQEProblem

Class functionality:

As described in the Qrisp documentation:

"Central structure to facilitate treatment of VQE problems. This class encapsulates the Hamiltonian, the ansatz, and the initial state preparation function for a specific VQE problem instance."

Class constructor and attributes:

The constructor receives the following parameters:

- 'hamiltonian: QubitOperator | FermionicOperator' – the problem Hamiltonian. If a 'FermionicOperator' is supplied it is automatically converted to a 'QubitOperator'. The Hamiltonian is hermitized, ladder-conjugate-reduced and threshold-filtered before storage.

- 'ansatz_function: callable' – a function receiving a 'QuantumVariable' and an array of real parameters; it applies one layer of the variational ansatz.

- 'num_params: int' – the number of free parameters per ansatz layer.

- 'init_function: callable = None' – optional function that prepares the initial state from |0⟩.

- 'callback: bool = False' – if 'True', the cost value of each optimisation step is appended to 'optimization_costs'.

All parameters are instantiated as class attributes. Two additional attributes track callback data:

- 'optimization_params: List<Object>' – initialised as an empty list; currently never populated (only 'optimization_costs' is actively filled).

- 'optimization_costs: List<float> | List<jax.Array>' – stores the cost values of each optimisation step when callback is active.

Class methods:

- 'set_callback' – sets 'callback = True'.

- 'set_init_function' – replaces the stored 'init_function'.

- 'compile_circuit' – compiles the parametrised VQE circuit for a given 'QuantumVariable' and depth and returns the 'QuantumCircuit' together with the list of 'sympy.Symbol' parameters.

- 'optimization_routine' – runs the classical parameter optimisation loop given a state-preparation callable and measurement keyword arguments.

- 'run' – executes the full VQE workflow for a given state-preparation callable, depth, and optional solver settings; returns the optimised energy as a float or jax.Array (in tracing mode).

- 'train_function' – runs the same optimisation as 'run', but instead of returning the energy it returns a callable that can be applied to a 'QuantumVariable' to prepare the optimised ground state.

- 'benchmark' – sweeps over depth, precision, and iteration ranges and returns a 'VQEBenchmark' object.

- 'visualize_energy' – plots the energy convergence curve; optionally overlays the exact ground-state energy.

Details and design decisions:

The class does not inherit from any Qrisp class. It has an aggregation to 'QubitOperator' (the stored Hamiltonian) because the operator can exist independently. The callable attributes 'ansatz_function' and 'init_function' are treated as first-class objects and are not shown as separate classes in the diagram.

VQEBenchmark

Class functionality:

As described in the source code: "This class is a wrapper for representing and evaluating the data collected in the benchmark method." It stores the raw performance measurements from multiple VQE runs and exposes analysis utilities.

Class constructor and attributes:

The constructor receives a 'benchmark_data' dictionary and two additional parameters:

- 'optimal_energy: float' – the exact ground-state energy of the Hamiltonian, used as the reference for the approximation-ratio metric.

- 'hamiltonian: QubitOperator' – the problem Hamiltonian against which the runs were benchmarked.

The class attributes extracted from 'benchmark_data' are:

- 'layer_depth: List<int>' – VQE layer count for each run.

- 'circuit_depth: List<int>' – compiled circuit depth for each run.

- 'qubit_amount: List<int>' – qubit count of the compiled circuit for each run.

- 'precision: List<float>' – Hamiltonian evaluation precision for each run.

- 'iterations: List<int>' – number of backend calls for each run.

- 'energy: List<float>' – optimised energy value for each run.

- 'runtime: List<float>' – wall-clock time in seconds for each run.

- 'optimal_energy: float' – as described above.

- 'hamiltonian: QubitOperator' – as described above.

Class methods:

- 'evaluate' – evaluates the collected data with a configurable cost metric (default: overall quantum volume) and gain metric (default: approximation ratio); returns two lists of floats (List<float>, List<float>).

- 'visualize' – plots a cost-vs-gain scatter diagram using the selected metrics.

- 'rank' – sorts the runs by a chosen metric; supports averaging across repeated runs with identical parameters ('average_repetitions=True') and optional table printing ('print_res=True').

- 'print_rank_table' – prints a formatted table of ranked run data.

- 'save' – serialises the benchmark object to a file using 'dill'.

- 'load' – class method that deserialises a 'VQEBenchmark' object from a file.

Details and design decisions:

The class does not inherit from any Qrisp class. It has an aggregation to 'QubitOperator', as the Hamiltonian can exist independently of the benchmark wrapper.

QAOAProblem

Class functionality:

As described in the Qrisp documentation: "Central structure to facilitate treatment of QAOA problems. This class encapsulates the cost operator, mixer operator, and classical cost function for a specific QAOA problem instance."

Class constructor and attributes:

The constructor receives the following parameters:

- 'cost_operator: callable' – a function that receives a 'QuantumVariable' or 'QuantumArray' and a parameter γ and applies the cost operator.

- 'mixer: callable' – a function that receives a 'QuantumVariable' or 'QuantumArray' and a parameter β and applies the mixer operator.

- 'cl_cost_function: callable' – the classical cost function that takes a dictionary of measurement results as input.

- 'init_function: callable = None' – optional function for preparing the initial state; default is the uniform superposition |+⟩n.

- 'callback: bool = False' – if 'True', intermediate optimisation results are stored.

All parameters become class attributes. Two additional callback attributes are initialised as empty lists: 'optimization_params: List<Object>' (currently never populated – only 'optimization_costs' is actively filled) and 'optimization_costs: List<Float>'.

Class methods:

- 'set_callback' – sets 'callback = True'.

- 'set_init_function' – replaces the stored 'init_function'.

- 'computeParams' – computes TQA warm-start initial angles γ and β for a given number of partitions and time step; returns a concatenated ndarray | jax.Array.

- 'compile_circuit' – compiles the parametrised QAOA circuit for a given 'QuantumVariable' or 'QuantumArray' and depth and returns the compiled 'QuantumCircuit' with parameter symbols.

- 'optimization_routine' – runs the classical optimisation loop.

- 'run' – executes the full QAOA workflow and returns the measurement results as a dict or jax.Array (in tracing mode).

- 'train_function' – runs the same optimisation as 'run', but returns a callable that applies the optimised QAOA circuit in-place to a given 'QuantumVariable', instead of returning measurement results.

- 'benchmark' – sweeps over depth, shot, and iteration ranges and returns a 'QAOABenchmark' object.

- 'visualize_cost' – plots the cost convergence curve from the callback data.

Details and design decisions:

The class does not inherit from any Qrisp class. It is the parent class of 'QIROProblem'. All operator arguments are callables stored as plain attributes.

QAOABenchmark

Class functionality:

As described in the source code: "This class is a wrapper for representing and evaluating the data collected in the benchmark method." It is the QAOA counterpart to 'VQEBenchmark'.

Class constructor and attributes:

The constructor receives a 'benchmark_data' dictionary, an 'optimal_solution' and a 'cost_function' callable. The class attributes extracted from the dictionary are:

- 'layer_depth: List<int>' – QAOA layer count for each run.

- 'circuit_depth: List<int>' – compiled circuit depth for each run.

- 'qubit_amount: List<int>' – qubit count for each run.

- 'shots: List<int>' – number of shots per backend call for each run.

- 'iterations: List<int>' – number of backend calls for each run.

- 'counts: List<Map<String, Float>>' – measurement results of the optimised circuit for each run, as a probability distribution mapping bitstrings to floats.

- 'runtime: List<float>' – wall-clock time in seconds for each run.

- 'optimal_solution' – the known optimal solution; type depends on the problem.

- 'cost_function: callable' – the classical cost function of the benchmarked problem.

Class methods:

- 'evaluate' – evaluates performance using configurable cost and gain metrics (default: overall quantum volume and approximation ratio); returns two lists (List<int> | List<float>, List<float>).

- 'visualize' – plots a cost-vs-gain scatter diagram.

- 'rank' – sorts the runs by a chosen metric; supports averaging across repeated runs with identical parameters ('average_repetitions=True') and optional table printing ('print_res=True').

- 'print_rank_table' – prints a formatted ranking table.

- 'save' – serialises the object to a file using 'dill'.

- 'load' – class method that deserialises a 'QAOABenchmark' object from a file.

Details and design decisions:

The class does not inherit from any Qrisp class. The 'cost_function' callable is stored as an attribute; it exists independently, hence it is an aggregation.

QIROProblem

Class functionality:

As described in the source code: "Central structure to run QIRO algorithms. A subclass of the QAOAProblem class. The idea is based on the paper by J. Finzgar et al." QIRO (Quantum-Informed Recursive Optimization) extends QAOA by iteratively reducing the problem graph based on quantum correlation measurements.

Class constructor and attributes:

The constructor receives the following parameters:

- 'problem: Any' – the problem structure (e.g., a NetworkX graph for MaxClique); a deep copy is stored.

- 'replacement_routine: callable' – function that adjusts the problem after the highest correlation value is identified.

- 'cost_operator: callable' – function that receives a single list '[problem, solutions, exclusions]' and returns a cost-operator function.

- 'mixer: callable' – function that receives a single list '[problem, solutions, exclusions]' and returns a mixer function.

- 'cl_cost_function: callable' – the classical cost function for the problem.

- 'init_function: callable' – a factory callable. In the constructor it is called with no arguments – 'init_function()' – to produce the default initial-state preparation function. During 'run_qiro' it is called with keyword arguments 'solutions' and 'exclusions' to produce an updated preparation function for each recursion step.

- 'revert: bool = False' – accepted by the constructor but not stored or used; likely a stub for future functionality.

The constructor calls the 'QAOAProblem' parent constructor with the initial operators derived from the problem. Additional unique class attributes are:

- 'problem: Any' – a deep copy of the problem structure; updated in-place by the replacement routine after each recursion.

- 'qiro_cost_operator: callable' – the QIRO-specific cost operator generator.

- 'qiro_mixer: callable' – the QIRO-specific mixer generator.

- 'replacement_routine: callable' – as described above.

- 'qiro_init_function: callable' – the stored init-function generator.

Class methods:

- 'run_qiro' – executes the full QIRO workflow: runs an initial QAOA pass, then for 'n_recursions' iterations calls the replacement routine (which identifies the highest-correlation assignment and updates the problem), updates the cost operator, mixer, and init function accordingly, and runs QAOA again. Returns the final measurement result as a dict or jax.Array (in tracing mode).

Details and design decisions:

The class inherits from 'QAOAProblem' and extends it with additional mutable state: the problem structure and the QIRO-specific callable generators. The 'cost_operator', 'mixer', and 'init_function' attributes inherited from 'QAOAProblem' are replaced on every recursion step during 'run_qiro'.

DCQOProblem

Class functionality:

As described in the source code: "General structure to formulate Digitized Counterdiabatic Quantum Optimization problems. This class is used to solve DCQO problems with the algorithms COLD (counterdiabatic optimized local driving) or LCD (local counterdiabatic driving)."

Class constructor and attributes:

The constructor receives the following parameters (all are instantiated as class attributes):

- 'Q: ndarray' – the QUBO matrix.

- 'H_init: QubitOperator' – the initial Hamiltonian at t = 0.

- 'H_prob: QubitOperator' – the problem Hamiltonian at t = T.

- 'A_lam: QubitOperator | List<QubitOperator>' – operator approximating the adiabatic gauge potential (AGP).

- 'agp_coeffs: callable' – the AGP coefficients function; called as agp_coeffs(lam) for LCD and as agp_coeffs(lam, f, f_deriv) for COLD, where f and f_deriv are the control-pulse values.

- 'lam_func: callable' – scheduling function λ(t, T) mapping t ∈ [0, T] to λ ∈ [0, 1]; must return a sympy expression.

- 'g_func: callable = None' – inverse of λ(t, T); required only for COLD.

- 'H_control: QubitOperator = None' – control-pulse Hamiltonian for COLD; if omitted, LCD is used.

- 'qarg_prep: callable = None' – optional initial-state preparation function; if omitted, the ground state of the X operator |−⟩n is prepared by default.

Two further attributes are derived inside the constructor:

- 'J: ndarray' – 0.5 × Q (coupling matrix).

- 'h: ndarray' – bias vector derived from Q, computed as -0.5 * diag(Q) - 0.5 * rowsum(Q).

Two placeholder attributes are initialised to 'None' and populated lazily by '_precompute_timegrid':

- 'lam: ndarray | None' – λ values on the time grid.

- 'lamdot: ndarray | List<Float> | None' – time-derivative λ̇ values.

Class methods:

- '_precompute_timegrid' – computes the λ and λ̇ arrays for a given number of time steps, evolution time T, and algorithm method (LCD or COLD).

- '_precompute_opt_pulses' – precomputes the sin/cos pulse basis-function matrices used by the COLD optimisation.

- 'apply_lcd_hamiltonian' – applies the LCD Hamiltonian evolution to a 'QuantumVariable'.

- 'apply_cold_hamiltonian' – applies the COLD Hamiltonian evolution to a 'QuantumVariable'.

- 'compile_U_cold' – compiles the COLD unitary circuit.

- 'optimization_routine' – runs the classical optimisation loop for COLD pulse parameters; supports multiple objective functions ("exp_value" or "agp_coeff_magnitude").

- 'QUBO_cost' – evaluates the QUBO cost xT Q x for a given measurement result.

- 'run' – executes the full DCQO workflow (LCD or COLD) and returns a dictionary mapping bitstrings to [probability, QUBO cost] pairs.

Details and design decisions:

The class does not inherit from any Qrisp class. It has aggregations to 'QubitOperator' for 'H_init', 'H_prob', 'H_control', and to 'QubitOperator | List<QubitOperator>' for 'A_lam', since these operators can exist independently. The callable attributes are plain Python functions stored as first-class objects.

QuantumBacktrackingTree

Class functionality:

As described in the Qrisp documentation: "This class describes the central data structure to run backtracking algorithms in a quantum setting. Backtracking algorithms are a very general class of algorithms which cover many problems of combinatorial optimization such as 3-SAT or TSP."

The quantum version of backtracking is based on the algorithm proposed by Montanaro and achieves a quadratic speed-up over its classical counterpart. A backtracking problem is specified by a maximum recursion depth and two predicate functions – 'accept' and 'reject' – each returning a 'QuantumBool'.

Node encoding: The height of a node (leaves have height 0, the root has height max_depth) is stored in a one-hot 'OHQInt' attribute 'h'. The path from the root is stored in reverse order in a 'QuantumArray' attribute 'branch_qa'.

Class constructor and attributes:

The constructor receives:

- 'max_depth: int' – the maximum recursion depth of the tree.

- 'branch_qv: QuantumVariable' – a template 'QuantumVariable' whose size determines the branching degree (2size branches per node).

- 'accept: callable' – a function receiving the tree and returning a 'QuantumBool' indicating whether the current node satisfies the acceptance criterion.

- 'reject: callable' – a function receiving the tree and returning a 'QuantumBool' indicating whether the current branch should be pruned.

- 'subspace_optimization: bool = False' – enables a circuit optimisation that exploits the structure of the non-algorithmic subspace to reduce gate count.

The class attributes are:

- 'max_depth: int' – as above.

- 'degree: int' – the branching degree, equal to 2(size of branch_qv).

- 'branch_qa: QuantumArray' – the reversed path register of length 'max_depth'.

- 'h: OHQInt' – the one-hot height register of size 'max_depth + 1'.

- 'qs: QuantumSession' – the session shared by 'h' and 'branch_qa'.

- 'accept_function: callable' – the stored accept predicate.

- 'reject_function: callable' – the stored reject predicate.

- 'subspace_optimization: bool' – as above.

Class methods:

- 'accept' – calls 'accept_function(self)' and returns its 'QuantumBool' result.

- 'reject' – calls 'reject_function(self)' and returns its 'QuantumBool' result.

- 'qstep_diffuser' – performs the quantum walk operator RA or RB (diffuser) for nodes of even or odd height.

- 'quantum_step' – applies one complete quantum walk step (RA followed by RB) to the tree state.

- 'estimate_phase' – performs quantum phase estimation on the quantum walk operator and returns a 'QuantumFloat' holding the phase estimate.

- 'init_phi(path: List<Integer>)' – initialises the tree into the equal superposition Φ state used by the quantum walk.

- 'init_node' – initialises the tree into the state corresponding to a specific path from the root.

- 'subtree' – returns a 'Subtree' object representing the subtree rooted at the given path.

- 'copy' – returns a deep copy of the tree as a 'Subtree' object.

- 'find_solution' – classically traverses the tree by repeatedly calling the quantum algorithm to determine which child node contains a solution; accepts an optional classical cl_accept predicate and measurement kwargs; returns the solution path as a List<Integer>.

- 'path_decoder' – decodes a measured height and branch-array state into a human-readable path list.

- 'statevector' – returns a SymPy quantum state representing the tree's statevector with decoded node labels.

- 'statevector_graph(return_root: bool = False)' – returns a NetworkX DiGraph representing the quantum state as a tree; nodes carry a complex amplitude attribute. If return_root=True, returns a tuple (DiGraph, QBTNode) where the second element is the root node.

- 'visualize_statevector' – plots the statevector graph using matplotlib.

Details and design decisions:

The class does not inherit from any Qrisp class. It has a composition to 'QuantumArray' ('branch_qa') and to 'OHQInt' ('h'), since both quantum registers only exist in the context of a specific tree instance. The 'QuantumSession' is an aggregation, as it can exist independently. The callable predicates 'accept_function' and 'reject_function' are stored as plain Python functions.

Supporting Classes

Subtree

Class functionality:

'Subtree' is a subclass of 'QuantumBacktrackingTree' that represents a subtree of a larger tree rooted at a specific path. It allows the quantum backtracking algorithm to focus on a subset of the search space from a given node downward.

Class constructor and attributes:

The constructor receives:

- 'parent_tree: QuantumBacktrackingTree' – the full tree from which this subtree is extracted.

- 'root_path: List<Integer>' – the path from the root of 'parent_tree' to the root of this subtree.

The parent constructor is called internally to set up the standard 'QuantumBacktrackingTree' attributes, re-using the same accept/reject callables and depth settings. Additional class attributes set during construction:

- 'root_path: List<Integer>' – as above.

- 'original_tree: QuantumBacktrackingTree' – reference to 'parent_tree'; set internally, not passed as a parameter.

- 'max_depth: int' – adjusted depth: parent_tree.max_depth - len(root_path).

Class methods:

- 'init_node(path)' – initialises the subtree to the node at root_path + path: sets 'h' to max_depth - len(path) and writes the combined path into 'branch_qa' in reverse order (path[::-1]).

- 'init_phi(path)' – initialises the Φ superposition state: builds 'h_state' with amplitude √max_depth at height 'max_depth' and alternating signs (-1)i at lower heights, then conditionally encodes 'root_path' and 'path' into 'branch_qa' for each height level.

- 'subtree(path: List<Integer>)' – delegates to 'original_tree.subtree(path)' and returns a 'Subtree'. Note: the current 'root_path' offset is not applied — the path is always interpreted relative to the top-level tree's root.

Details and design decisions:

'Subtree' inherits from 'QuantumBacktrackingTree' and overrides the initialisation methods to account for the offset caused by the non-root starting position. The 'original_tree' reference allows the subtree to delegate to the top-level tree's registers where appropriate.

QBTNode

Class functionality:

'QBTNode' is a classical Python object that represents a single node in a 'QuantumBacktrackingTree'. It is used during the classical traversal phase ('find_solution') to identify which child node contains a solution, based on quantum amplitude estimation.

Class constructor and attributes:

The constructor receives:

- 'tree: QuantumBacktrackingTree' – the tree to which this node belongs.

- 'path: List<Integer>' – the path from the root to this node as a list of branch indices.

- 'amplitude: numpy.complex128 = None' – the estimated quantum amplitude associated with this node.

The class attributes are:

- 'h: int' – the height of the node, computed internally as tree.max_depth - len(path) (0 = leaf, max_depth = root).

- 'path: List<Integer>', 'tree', 'amplitude: numpy.complex128' – as above.

Class methods:

- 'sv_specifier' – returns a dictionary mapping quantum registers to classical values encoding this node's position: tree.h → node height; each entry of tree.branch_qa (in reversed index order) → the corresponding branch decision, padded with 0 for levels beyond the path length. If the node belongs to a 'Subtree', 'root_path' is prepended to the local path to form the global path before encoding.

- '__hash__' – hashes the node based on str(self.path) only; 'tree' and 'amplitude' are ignored, so nodes from different trees with the same path will collide.

- '__str__' – returns "QBTNode(path = ..., amplitude = ...)".

- '__repr__' – delegates to '__str__'.

Details and design decisions:

'QBTNode' is a plain Python class with no Qrisp inheritance. The 'tree' reference is an aggregation. Instances are created transiently inside 'statevector_graph' to represent nodes in the statevector DiGraph; they are not stored in the tree itself.
