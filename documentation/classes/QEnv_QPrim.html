QEnvironment and QPrimitive

Information about classes and methods

The ‘Environment’ diagram is named after the central class of the diagram. It unites the ‘QuantumPrimitive’ with its descendants, outlining the class hierarchy. Different implementations of ‘QuantumEnvironments’ can be found in this diagram.

QuantumPrimitive

Class functionality:

This class provides a wrapper class to identify Qrisp primitives. It is on top of the class hierarchy of the ‘Environment’.

Class constructor and attributes:

The class constructor has one parameter that receives the ‘name’ as a string. The constructor is calling the parent constructor of the base class ‘Primitive’. The ‘name’ parameter is instantiated as ‘qrisp_name’.

Details and design decisions:

The class does not have methods. Even though it is stated in the description that it is on top of the class hierarchy, this only means the top of the Qrisp hierarchy part of this diagram. The top of this diagram is the ‘Primitive’ class, ‘QuantumPrimitive’ inherits from it.

OperationPrimitive (not in 0.8)

Class functionality:

‘OperationPrimitive’ wraps a quantum operation object and exposes it as a ‘QuantumPrimitive’. This allows the operation to be used within a JAX-based quantum framework that distinguishes between abstract evaluation.

Class constructor and attributes:

The class constructor has one parameter that receives the operation, to be wrapped. The attribute is called ‘op’.

Class methods:

The class methods are:

  • ‘inverse’ – this method returns a ‘OperationPrimitive’ with an inversed operation.
  • ‘control’ – this method returns a ‘OperationPrimitive’ with a controlled operation.

Details and design decisions:

The class inherits from ‘QuantumPrimitive’. It has an aggregation to the Operation class, which is described in in the ‘Operation’ diagram.

QuantumEnvironment

Class functionality:

As described in the source code [14]:

  “QuantumEnvironments are blocks of code, that undergo some user-specified compilation process. They can be entered using the ``with`` statement”

Class constructor and attributes:

The class constructor receives ‘env_args’, which can be any type of argument. It calls the parent constructor of ‘QuantumPrimitive’ with the name ‘q_env’. It also sets the ‘multiple_results’ attribute to true and defines a decorator function ‘abstract_eval’. Finally the ‘env_args’ parameter is getting instantiated. The class attributes are:

  • ‘deepest_environment: List[QuantumEnvironment]’ – This attribute is static and describes the deepest environment
  • ‘env_args: List[Object]’ – This attribute is a list of arguments to be used by the class
  • ‘env_qs: QuantumSession’ – The ‘QuantumSession’ of the ‘QuantumEnvironment’
  • ‘original_data: List[Instructions]’ – “This list stores the original data of the quantum session tracked” [14]
  • ‘env_data: List[Instruction]’ – “[…] which is the list of instructions, that have been appended in this environment. Note that child environments append themselves in this list upon exiting.” [14]
  • ‘deallocated_qubits: List[Qubit]‘ - „This list stores the qubits that have been deallocated in this environment. This information is required because the need to be temporarily reallocated to prevent compilation errors at compile time.” [14]
  • ‘temp_qubit_cache: Map[Integer, Qubit]’ – A dictionary required for checking the tracing mode
  • ‘parent: QuantumEnvironment’ – this is the deepest environment
  • ‘active_qs_list: List[QuantumSession]’ – "Acquire a list of all active quantum sessions" [14]
  • ‘manual_allocation_management : bool’ – "Manual allocation management means that the compile method can process allocation and deallocation gates." [14]

Class methods:

The class methods are as follows:

  • 'abstract_eval' – this class is only internally and thus private, it evaluates an 'AbstractQuantumCircuit' object. It is annotated with jax_rule, to indicate that the JAX framework is utilizing this method.
  • 'start_dumping'- "The methods to start the dumping process for this environment." [14]
  • 'stop_dumping' – this method ends the dumping process for this environment
  • '__enter__' – this dunder method is called as it enters the environment.
  • '__exit__' - this dunder method is called as it exits the environment.
  • 'compile' - "This is the default compilation method. It simply compiles all subenvironments and the collected data to the session.[…]" [14]
  • 'jcompile' – The method evaluates a JAX intermediate representation and writes the resulting values back into the compilation context

Details and design decisions:

The class is an environment and thus implements the dunder methods required for supplying the environment functionality, they are marked with ‘<’. Furthermore this class uses the jax library, instantiating a method in the class constructor for utilization. The class has many associations of which 2 are aggregations: the ‘parent’ and the ‘active_qs_list’, as they exist outside the ‘QuantumEnvironment’. The other attributes are compositions as they only exist during the lifetime of a ‘QuantumEnvironment’ object. The diagram shows question marks at the '__exit__' methods parameters, as these parameters are never used and thus not determinable. The class inherits from the 'QuantumPrimitive' class and is the parent class of the following classes:

  • CLControlEnvironment
  • ConjugationEnvironment
  • ControlEnvironment
  • CustomControlEnvironment
  • GateWrapEnvironment
  • GMSEnvironment
  • IterationEnvironment
  • JIterationEnvironment
  • ConditionEnvironment
  • InversionEnvironment

ClControlEnvironment

Class functionality:

As described in the source code [14]:

  “The ‘ClControlEnvironment’ enables execution of quantum code conditioned on classical values. The environment works with similar semantics as the ‘ControlEnvironment’, implying this environment can also be entered using the ‘control’ keyword.”

Class constructor and attributes:

The class constructor receives ‘ctrl_bls’, which is a list of bools. It also receives a ‘ctrl_state’ integer, which is -1 by default. Finally the constructor receives a bool ‘invert’, which is by default false, all of them get instantiated as class attributes.

Class methods:

The class methods are as follows:

  • 'compile' – This compile method calls the ‘QuantumEnvironment’ compile method only when the corresponding ‘ctrl_bls’ match the ‘ctrl_state’
  • '__exit__' - this dunder method is called as it exits the environment.
  • 'jcompile' – The method evaluates a JAX intermediate representation and writes the resulting values back into the compilation context

Details and design decisions:

The class is an environment and thus implements the dunder methods required for supplying the environment functionality, they are marked with ‘<’. The class inherits from the 'QuantumEnvironment’ class.

qRange

Class functionality:

The ‘qRange’ class implements a reversible ‘QuantumFloat’ iterator, that works with integers. It implements a quantum-controlled version of Python’s ‘range()’ function.

Class constructor and attributes:

The class constructor for ‘qRange’ receives and instantiates a ‘QuantumFloat’ ‘max_index_qf and an optional bool ‘create_index_qf’, which is false by default.

The class attributes are:

  • index_qf: QuantumFloat – this attribute functions as a quantum loop counter register and drives the iteration in a reversible and unitary way
  • quantum_condition: ClControlledEnvironment – this environment conditionally executes the loop body based on the sign bit of the ‘index_qf’ register. It implements a reversible quantum loop condition
  • c_index: int – this integer is a loop counter that track the current iteration index is is returned on each __next__ method call

Class methods:

The class methods are as follows:

  • _iter_(): qRange – this dunder method prepares the iterator object and returns it
  • _next_(): int – this dunder method implements the logic of the next step of the iterator

Details and design decisions:

The ‘qRange’ class has two dunder methods that are labelled as ‘< ’. Furthermore the class holds a composition for the ‘quantum_condition_env‘ attribute.

ConjugationEnvironment

Class functionality:

As described in the qrisp documentation [25]:

“This QuantumEnvironment can be used for perfoming conjugated operations. An arbitrary unitary can be conjugated by another unitary :

This structure appears in many quantum algorithms such as Grover, Quantum backtracking or Fourier arithmetic.

Using the ConjugationEnvironment not only helps to structure the code, but can also grant performance advantages.

This is because the controlled circuit of such a conjugation is can be realized by just controlling instead of all three operations.

The ConjugationEnvironment can be called using the alias conjugate. Conjugate takes the conjugation function (in our example ) and returns a function that takes the arguments for the conjugation function and returns the corresponding ConjugationEnvironment. For more information consult the examples section.”

Class constructor and attributes:

The class constructor receives a function ‘conjugation_function’, which is “the function performing the operation V” [25]. It also receives an iterable of objects ‘args’, which are “the arguments for the conjugation function” [25]. Furthermore it receives a mapping from a string to an object ‘kwargs’, for “the keyword arguments for the conjugation function” [25]. Also the constructor receives a bool ‘allocation_management’, which is by default true. All of them get instantiated as class attributes.

The class has one more attribute that is set in the ‘enter’ method: ‘conjugation_circ’. This is a copy ‘QuantumCircuit’ of the ‘env_qs’ that is used for the conjugation.

Class methods:

The class methods are as follows:

  • ‘__enter__’ – This dunder method is called as it enters the environment. It mostly prepares data for the conjugation.
  • '__exit__' - this dunder method is called as it exits the environment. It performs the conjugation or performs an inversion if in tracing mode.
  • ‘perform_conjugation’ – this is the logic of the conjugation.
  • ‘jcompile’ - This method implements the JAX compilation logic
  • ‘compile’ – This method implements the compilation logic

Details and design decisions:

The class is an environment and thus implements the dunder methods required for supplying the environment functionality, they are marked with ‘<’. Furthermore this class uses the jax library. The method ‘compile’ uses a parameter ctrl, which is never used, so it has a ‘?’ type. The ‘perform_conjugation’ method uses the decorator ‘@custom_control’, it is labelled with ‘<’. The class has an composition to the ‘QuantumCircuit’ class, since the circuit wouldn’t exist without the ‘ConjugationEnvironment’. The class inherits from the 'QuantumEnvironment’ class.

ControlEnvironment

Class functionality:

As described in the Qrisp Documentation of ‘ControlEnvironment’ [26]:

“This class behaves similarly to ConditionEnvironment but instead of a function calculating a truth value, we supply a list of qubits. The environment’s content is then controlled on these qubits.

An alias for this QuantumEnvironment is “control”.“

Class constructor and attributes:

The class constructor receives following parameters:

  • ‘ctrl_qubits: List[Qubit]’ – this is the list of qubits on which to control the environments contents. [26]
  • ‘ctrl_state: int | String = -1’ – “The computational basis state which is supposed to activate the environment. Can be supplied as a bitstring or integer. The default is “1111..”.” [26]
  • ‘ctrl_method: String = null’ – this is the method being passed to the controlled Instruction in the compiler
  • ‘invert: bool = false’ – this bool decides over control states

All parameters get instantiated as class attributes. Furthermore the remaining class attributes are:

  • ‘arg_qs: QuantumSession’ – these are the merged sessions of the ‘ctrl_qubits’
  • ‘sub_condition_envs: List[ ControlEnvironment | ConditionEnvironment]’ - This attrbute prevents redundant application of control gates by allowing the parent environment to distinguish between normal quantum gates and nested environments.
  • ‘condition_truth_value: Qubit’ – this qubit represents if the ‘ControlEnvironment’ is active or true
  • ‘qbool: QuantumBool’ – this is a ancilla qubit holding the environments condition truth value
  • ‘parent_cond_env: ControlEnvironment | ConditionEnvironment’ – this attribute is needed to make nested control or condition blocks compile correctly and efficiently

Class methods:

The class methods are as follows:

  • ‘__enter__’ – This dunder method is called as it enters the environment. It mostly prepares data. It returns the ‘condition_truth_table’
  • '__exit__' - this dunder method is called as it exits the environment. It determines the tracing mode and then the parent environment
  • ‘compile’ – This method implements the compilation logic
  • ‘jcompile’ - This method implements the JAX compilation logic

Details and design decisions:

The class is an environment and thus implements the dunder methods required for supplying the environment functionality, they are marked with ‘<’. The ‘sub_condition_envs’ and ‘parent_cond_env’ are in round brackets to imply that it can be either one of them and that the attribute doesn’t have to be of that specific type but can. The class has a composition to ‘Qubit’, because the ‘ctrl_qubits’ wouldn’t serve a purpose without a context, same goes for the ‘QuantumBool’ association and the ‘sub_condition_envs’ attribute. The ‘QuantumSession’ ‘arg_qs’, the ‘parent_cond_env’ and ‘qbool’ exist on their own, thus they are aggregations. The class inherits from ‘QuantumEnvironment’.

CustomControlEnvironment

Class functionality:

The purpose of ‘CustomControlEnvironment’ is to rewrite all instructions in an environment into ‘CustomControlOperation’, controlled by a specific qubit. Most importantly this is happening without using standard controlled-gate synthesis.

Class constructor and attributes:

The class constructor has two parameters, of which only one is used: ‘control_qb’, it is instantiated as a Qubit. It is the control qubit that controls the operations.

Class methods:

The class methods are as follows:

  • ‘__enter__’ – This dunder method is called as it enters the environment. The method calls the parents ‘__enter__’ method and merges the environment’s ‘QuantumSession’ with the ‘control_qb’s ‘QuantumSession’.
  • ‘compile’ - This method implements the compilation logic

Details and design decisions:

The class is an environment and thus implements the dunder methods required for supplying the environment functionality, they are marked with ‘<’. This method has an aggregation to the ‘Qubit’ class, because the Qubit is already preexisting to the ‘CustomControlEnvironment’. The class inherits from the ‘QuantumEnvironment’ class.

GateWrapEnvironment

Class functionality:

As described in the Qrisp documentation:
“This environment allows to hide complexity in the circuit visualisation. Operations appended inside this environment are bundled into a single Instruction object.

The functionality of this Quantum Environments can also be used with the gate_wrap decorator.

After compiling, the wrapped instruction can be retrieved using the .instruction attribute.“ [27]

Class constructor and attributes:

The class constructor calls the parent constructor and instantiates the ‘name’ of the class as a String, by default null. It then sets the parents class attribute ‘manual_allocation_management‘ to true. The class has one more unique attribute: ‘instruction’, which is (obviously) an ‘Instruction’.

Class methods:

The class has a single method:
‘compile’ – this method implements the compilation logic.

Details and design decisions:

The class has a single composition, the ‘instruction’. This instruction only exists during the lifetime of a ‘GateWrapEnvironment’ object. The class inherits from ‘QuantumEnvironment’.

GMSEnvironment

Class functionality:

As described in the Qrisp documentation:

“This environment provides a convenient interface for constructing quantum algorithms using the Ion-trap native GMS gates. GMS gates allow entangling more than 2 qubits in a single step and can therefore boost performance in many situations. For more information on GMS gates consult https://arxiv.org/abs/quant-ph/9810040. The techniques for converting the circuits presented to this environment are mostly based on https://ieeexplore.ieee.org/document/9815035. This environment allows to code blocks of phase-only gates as we are used to but compiles these blocks to GMS gates.” [28]

Class methods:

The class has only one method:

‘compile’ – this method implements the compilation logic of the ‘GMSEnvironment’.

Details and design decisions:

This class has no unique attributes and inherits from the ‘QuantumEnvironment’.

IterationEnvironment

Class functionality:

As described in the Qrisp documentation:

“This QuantumEnvironment can be used for reducing bottlenecks in compilation time. Many algorithms such as Grover or QPE require repeated execution of the same quantum circuit. When scaling up complex algorithms that perform a lot of non-trivial logic many iterations can significantly slow down the compilation speed. The IterationEnvironment remedies this flaw by recording the circuit of a single iteration and then duplicating the instructions the required amount of times.

Another bottleneck that can appear is the compile method as the qubit allocation algorithm can also scale bad for really large algorithms. For this problem the IterationEnvironment exposes the precompile keyword. Setting this keyword to True will perform the Qubit allocation algorithm on the QuantumEnvironments content and then (if necessary) allocate another QuantumVariable to accomodate the workspace qubits of the compilation result. This way there is only a single (de)allocation per IterationEnvironment.“ [29]

Class constructor and attributes:

The class constructor receives a ‘QuantumSession’ ‘qs’, an integer ‘iteration_amount’ and a bool ‘precompile’, which is by default false. ‘iteration_amount’ and ‘precompile’ get instantiated as class attributes. Furthermore there is one class attribute ‘initial_qvs’, which is a the set of list of ‘QuantumVariables’. It is essentially the list of unique ‘QuantumVariables’ if the ‘qs’ ‘QuantumSession’.

Class methods:

The class methods are as follows:

  • ‘__enter__’ – this dunder method gets called when the code enters the ‘IterationEnvironment’. The method prepares the ‘inital_qvs’ parameter and calls the parent ‘__enter__’ method
  • ‘__exit__’ – this dunder method gets called when the code exists the ‘IterationEnvironment’
  • ‘compile’ – this method implements the compilation logic

Details and design decisions:

The class is an environment and thus implements the dunder methods required for supplying the environment functionality, they are marked with ‘<’. The class has a composition to ‘QuantumVariable’, because it only exists with an object of ‘IterationEnvironment’. The class inherits from ‘QuantumEnvironment’.

JIterationEnvironment

Class functionality:

The class only has the purpose of running a ‘iteration_env_evaluator’ function, which is defined in the ‘jiteration_environment.py’ class.

The idea is described in the source code [14]:

“The idea behind this equation compiler is the following: The jrange feature executes 2 iterations of the loop to capture the loop body in an QuantumEnvironment. We need 2 iterations to determine which inputs are updated each iteration and which inputs stay the same.”

Class methods:

As stated above, the class only has only one method ‘jcompile’, which executes the ‘Iteration_env_evaluator’.

Details and design decisions:

This class inherits from the class ‘QuantumEnvironment’. The types of the map couldn’t be determined.

JRangeIterator

Class functionality:

This class implements the logic, that is called in the function Jrange. The use is described in the source code [14]:

„Performs a loop with a dynamic bound. Similar to the Python native range this iterator can receive multiple arguments. If it receives just one, this value is interpreted as the stop value and the start value is assumed to be 0. Two arguments represent start and stop value, whereas three represent start, stop and step.“

Class constructor and attributes:

The class constructor receives a list of arguments, which can be of any type. The constructor instantiates the attributes ‘start’, ‘stop’ and ‘step’ depending on the count of arguments, if they get instantiated they are arrays of the type integer. The attributes of the class are as following:

  • ‘iteration: int’ – this is the iteration index, when it reaches 3 the Iterator ends
  • ‘loop_index: Array<Integer>’ – this the loop iteration index tracer
  • ‘iter_env: JIterationEnvironment’ - This is the ‘JIterationEnvironment’ in which the iteration logic happens
  • ‘iter_1_qvs: List<QuantumVariable>’- The ‘QuantumVariables’ of the ‘TracingQuantumSession’
  • ‘iter_2_qvs: List<QuantumVariable>’ – A second list of ‘QuantumVariables’ of the ‘TracingQuantumSession’

Class methods:

The class methods are as follows:

  • ‘_**iter_: JRangeIterator**’ – This dunder method makes the object iterable and return an iterator object
  • ‘_next_: Array<Integer>’ – this dunder method produces the next value in the iterator

Details and design decisions:

The ‘JRangeIterator’ class has a composition to ‘JIterationEnvironment’, as the environment only exists during the lifetime of a ‘JRangeIterator’ object. It also has a composition to QuantumVariable, for the same reasons.

ConditionEnvironment

Class functionality:

As described in the Qrisp documentation:

“This class enables the usage of if-conditionals as we are used to from classical programming” [30]

Class constructor and attributes:

The class constructor has 3 parameters: a callable function ‘cond_eval_function’, a list of objects ‘args’ and a map ‘kwargs’ of keyword arguments for ‘cond_eval_function’ [30]. The class instantiates all the parameters and merges the ‘args’ into a ‘QuantumSession’ ‘arg_qs’ and the ‘manual_allocation_management’ is set to true. The class attributes, excluding previously listed ones, are:

  • ‘sub_condition_envs: List<ContrlEnvironment | ConditionEnvironment>’ - This attribute is used for a list that will contain the qubits holding the truth values of the conditional/control environments within this environment. [30]
  • ‘qbool: QuantumBool’ - this is a ancilla qubit holding the environments condition truth value
  • ‘condition_truth_value: Qubit’ – this qubit represents if the ‘ConditionEnvironment’ is active or true
  • ‘parent_cond_evn: ControlEnvironment | ConditionEnvironment’ - this attribute is needed to make nested control or condition blocks compile correctly and efficiently

Class methods:

The class methods are as follows:

  • ‘__enter__’ – This dunder method is called as the ‘ConditionEnvironment’ is entered
  • ‘__exit__’ – This dunder method is called as the ‘ConditionEnvironment’ is exited
  • ‘compile’ – This method implements the compilation logic

Details and design decisions:

The class is an environment and thus implements the dunder methods required for supplying the environment functionality, they are marked with ‘<’. The ‘sub_condition_envs’ and ‘parent_cond_env’ are in round brackets to imply that it can be either one of them and that the attribute doesn’t have to be of that specific type but can. The class has a composition to ‘Qubit’, because the ‘ctrl_qubits’ wouldn’t serve a purpose without a context, same goes for the ‘QuantumBool’ association and the ‘sub_condition_envs’ attribute. The ‘QuantumSession’ ‘arg_qs’, the ‘parent_cond_env’ and ‘qbool’ exist on their own, thus they are aggregations. The class inherits from ‘QuantumEnvironment’.
