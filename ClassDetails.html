Explanation

In the following chapter the details of the depicted classes are explained. Furthermore, interesting functionalities, attributes, constructors, methods, details and design choices are disclosed.

Dictionaries are written as Maps, since they serve the same purpose and are more generally seen in UML class diagrams.

Sometimes datatypes are listed with logical OR Operations (|) – this implies that due to the dynamic typing of the eDSL there are multiple datatypes possible.

The dunder methods or decorators have <<>> descriptions in the diagram to describe the ground concept of the methods functionality.

Every class attribute that has already been described in the parent class will not be referenced in the child class in the diagrams.

In Python, class attributes describe what is in other statically typed object oriented languages considered to be a static method. When the documentations refers to class attributes it is refered to how classical object oriented languages define class attributes. Python class attributes are referred to as static methods, with an underlining.

Backends

Information about classes and methods

‘BackendClient’ and every subclass of it implements a ‘run’ method (except for AQTBackend), taking a ‘‘QuantumCircuit’’ as ‘qc’ and a number of ‘shots’. The ‘run’ method executes the ‘qc’, delivering measurement results in form of a dictionary, mapping a bitstring to an integer. Some classes have slight variations on parameters.

BackendClient

Class functionality:

‘BackendClients’ allows connecting to Qunicorn backend servers [6].

Class attributes and constructor:

An object of the class can be instantiated by an IP-Adress as a String ‘api_endpoint’ and optionally providing a ‘port’ integer and ‘token’ string.

Details and design decisions:

‘BackendClient’ is a central class in the Backends diagram, as many classes inherit from it.

VirtualBackend

Class functionality:

As the name suggests, ‘VirtualBackend’ provides a virtual backend for circuit execution. It runs on the same machine as a separate Python thread. The structure of it allows the easy setup of wrappers for foreign or modified circuits. [7].

Class attributes:

It contains a function ‘run_func’ that receives a ‘‘QuantumCircuit’’, an integer specifying the amount of shots and a token in the form of a string. It returns the counts as a dictionary of bitstrings [7].

It also contains a ‘backend_server’ , which is only instantiated if no port is specified in the constructor.

Class Constructor:

The constructor instantiates the ‘run_func’ attribute and receives an optional ‘port’ integer.

Details and design decisions:

‘VirtualBackend’ is inheriting the ‘BackendClient’.

BackendServer

Class functionality:

‘BackendServers’ allow the convenient setup of a server relating to the ‘Qunicorn’ interface [8].

Class attributes and constructor:

It contains a function ‘run_func’ that receives a ‘‘QuantumCircuit’’, an integer specifying the amount of shots and a token in the form of a string. It returns the counts as a dictionary of bitstrings [8].

An ‘ip_address’ is optionally set for the listening socket that should be opened. By default the IP-address of the host machine is used [8].

A ‘port’ can be entered optionally, by default it is 9010 [8].

A bool variable ‘is_simulator’ specifies whether the results returned by the server are due to a simulator. It is normally set to true [8].

A Flask object is instantiated as ‘app’. Flask provides a lightweight WSGI web application framework with the ability to scale up to complex applications [9]. The constructor implements methods for REST endpoints, that have not been included in the diagram.

A list of JSON objects are saved as ‘deployments’.

Finally a list of Maps which map Strings to any type named ‘jobs’.

Details and design decisions:

‘BackendServer’ is an attribute of ‘VirtualBackend’, that is only instantiated if a port is given.

QiskitBackend

Class functionality:

‘QiskitBackend’ instantiates a ‘VirtualBackend’ using a Qiskit backend. This allows easy accesss to Qiskit backends throught the qrisp interface [10].

Class constructor:

It is optionally instantiated with 2 parameters:
A ‘backend’ which is a AerSimulator object and a ‘port’, which is by default none [10]. The class constructor also defines the method ‘run’.

Details and design decisions:

The class does not have unique attributes and only implements the attributes from the parent class ‘BackendClient’.

QiskitRuntimeBackend

Class functionality:

“This class instantiates a VirtualBackend using a Qiskit Runtime backend. This allows easy access to Qiskit Runtime backends through the qrisp interface. It is imporant to close the session after the execution of the algorithm” [11].

Class constructor:
It is instantiated with an ‘api_token’ string, an optional ‘backend’, that can be either be a qiskit_ibm_runtime.Session or a qiskit.providers.BackendV2, an optional ‘channel’ String, which is “ibm_cloud” per default and an optional ‘mode’ String, being “job” per default. BatchedBackend

Class functionality and methods:

“This class tackles the problem that many physical backends have a high-overhead regarding individual circuit execution. This overhead typically comes from finite network latency, authentication procedures, compilation steps etc. Typically this overhead is remedied through supporting the execution of batches of circuits, which however doesn’t really fit that well into the Qrisp programming model, which shields the user from handling individual circuits and automatically decodes the measurement results into human readable labels.

In order to bridge these worlds and still allow automatic decoding, the BatchedBackend allows Qrisp users to evaluate measurements from a multi-threading perspective. The idea is here that the circuit batch is collected through several threads, which each execute Qrisp code until a individual backend call is required. This backend call is then saved until the batch is complete. The batch can then be sent through the .dispatch method, which resumes each thread to execute the post-processing logic”[12].

Class constructor:

For instantiation it receives a function ‘batch_run_func’, which is described in the documentation as: “A function that recieves a list of tuples in the form list[tuple[‘QuantumCircuit’, int]], which represents the quantum circuits and the corresponding shots to execute on the backend. It should return a list of dictionaries, where each dictionary corresponds to the measurement results of the appropriate backend call.” [12].

Class attributes:
‘batch_run_func’, which is the function instantiated in the constructor. ‘batch’, which is a list of tuples, that contain ‘QuantumCircuit’s and integers. ‘results_available’ as a bool and ‘result’, which is implemented as a map which maps an integer to an integer. ‘backend_exception’ is an attribute which is empty by default, if it is not empty, an exception will be thrown in the ‘run’ method.

Class methods:

The class implements the following methods:

  • ‘run’ – This method runs the the ‘BatchedBackend’ object.
  • ‘dispatched’ – “This method dispatches all collected queries and seubsequently resumes their threads.” [34]

Details and design decisions:

It inherits from ‘BackendClient’ and is the parent class of ‘AQTBackend’.

AQTBackend

Class functionality:

This class instantiates a ‘BatchedBackend’ using an AQT backend. This allows easy access to AQT backends through the qrisp interface. Using this backend requires addtional dependencies: pip install qiskit-aqt-provider. [13] (The documentation writes VirtualBackend, instead of BatchedBackend, which is false)

Class constructor:

The class can be instantiated with a ‘api_token’ string, a ‘device_instance’ string and a ‘workplace’ string. The constructor implements the method ‘run_batch_aqt’, which is used to instantiate the parent constructor. It is not listed in the diagram, as it only serves it’s function in the constructor.

Details and design decisions:

The class does not have unique attributes and only implements the attributes from the parent class ‘BatchedBackend’.

This concludes the ‘Backends’ UML class diagram.

Permeability

Information about classes and methods

The ‘Permeability’ diagram is a collection of classes that provide the ability to represent a permeability DAGs (Directed Acyclic Graph), describing a clear hierarchical execution of a task [17]. This facilitates numerous powerful transformations such as parallelization, memory management, synthesis of uncomputation and many more use-cases [18]. Permeability is a property of quantum gates, that was initially introduced by Frauenhofer Institute in [4] and gives a formal meaning of what “constant” could mean for a quantum function [18]. For a more detailed understanding of Permeability, here is is a Definition out of [4]:

Definition 2.1 (Permeability):

A (composite) quantum gate U is called Z-permeable in qubit i, if it commutes with the Z operator on this qubit.

U is permeable on qubit (1)

Similarly, U is called X-permeable in qubit i if it commutes with the X operator on this qubit

Operations

Information about classes and methods

The ‘Operations’ is one of the most central diagrams within the Qrisp framework. It is an especially important diagram as it defines the operations that can be applied to a quantum circuit. Central of this class is the ‘Operation’ class, practically every class inherits from it. Operations are used on a ‘‘QuantumCircuit’’ object, as quantum gates, measurements or classical logic gates [19].

Operation

Class functionality:

As described in the qrisp documentation:

“This class describes operations like quantum gates, measurements or classical logic gates. Operation objects do not carry information about which Qubit/Clbits they are applied to. This can be found in the Instruction class, which is a combination of an Operation object together with its operands.” [19]

Class constructor and attributes:

The class constructor is built with the following attributes, described in the Qrisp-documentation as:

“Operation objects consist of five basic attributes:

  • .name : A string identifying the operation
  • .num_qubits : An integer specifying the amount qubits on which to operate
  • .num_clbits : An integer specifying the amount of classical bits on which to operate.
  • .params : A list of floats specifying the parameters of the Operation
  • .definition : A ‘QuantumCircuit’. For synthesized (i.e. non-elementary) operations, this ‘QuantumCircuit’ specifies the operation.” [19]

Additionally an ‘init_op’ object is optionally received. Although these are the attributes received by the constructor the class has a few more attributes: ‘abstract_params’, which are represented as a list of Sympy Symbols [20], ‘is_qfree’ a bool describing if the unitary is a permutation matrix [14]. ‘permeability’ is a dict that maps integers, representing the indices of qubits to a bool, depending if it is permeable on that qubit. ‘Unitary’ is an ndarray, describing the unitary of the operation. The last available attribute is ‘lambdified_params’, which is a list of functions. ‘Unitary’ and ‘lambdified_params’ are both set within if-junctions and thus declared as “dynamically set” in an annotation within the diagram.

Class methods:

The class has many useful methods. The methods give possibilities to manipulate the Operation and to get further information, such as the inverse or unitary. A ‘Operation’ object can also be converted into a ‘ClControlledOperation’, with the method ‘c_if’.

Details and design decision:

The ‘Operation’ class is central of the diagram. With the ‘append’ method from ‘QuantumCircuit’ it is possible to apply these objects to a quantum circuit. A ‘QuantumCircuit’ can be converted to a ‘Operation’ object with the ‘to_gate’ function of the ‘QuantumCircuit’. Its sub classes are:

PTControlledOperation

Class functionality:

As described in the source code:

“This class describes phase tolerant controlled operations.

Phase tolerant means that the unitary can take the form

[D_0, 0, 0, 0]

[0, D_1,0, 0]

[0, 0, D_2, 0]

[0, 0, 0, U]

Where U is the operation to be controlled and D_i are diagonal operators.” [14]

Class constructor and attributes:

The class constructor has 4 parameters, an ‘Operation’ as ‘base_operation’, an optional integer ‘num_ctrl_qubits’ with 1 by default, an optional integer or String ‘ctrl_state’, which is by default -1 and an optional String or function ‘method’, which is “auto” by default. ‘num_ctrl_qubits’ is the range for the class attribute List ‘controls’. Every other parameter has a respective class attribute.

Class methods:

The class implements the methods from its parents class respectively to its ‘PTControlledOperation’ versions.

Details and design decision:

Two attributes of the class are surrounded by round brackets. Because this part of the code is unreachable, it was decided is not of relevance. The class ‘ControlledOperation’ inherits from this class.

CustomControlOperation

Class functionality:

The class extends Operations and creates a custom, control-style quantum operation based on an existing operation. It’s behavior depends on the flag ‘targeting_control’.

Class constructor and attributes:

The constructor has two cases depending on the bool ‘targeting_control’. When ‘targeting_control’ is false, the class adds an extra control qubit and shift the original operation onto the remaining qubits. If it is true, the operation is wrapped exactly as it is with no additional qubits or structural changes. The constructor also instantiates two attributes, that are unique to the class: the bool ‘targeting_control’ and the ‘init_op’ Operation.

Class methods:

The class has two methods, ‘inverse’ creating the inverse of the Operation and ‘copy’, creating a copy of the ‘CustomControlOperation’

Details and design decision:

This class inherits from ‘Operation’. There is also an aggregation, due to the ‘init_op’ object.

Information about classes and methods

The ‘QuantumMangagement’ diagram is a very important diagram within the Qrisp framework. QuantumCircuit

Class functionality:

The ‘QuantumCircuit’ class in Qrisp is used to describe and construct quantum circuits.
Its design and naming closely follow Qiskit’s ‘QuantumCircuit’, enabling strong interoperability. [23]

Class constructor and attributes:

The class instantiates 2 parameters. The parameters ‘num_qubits’ and ‘num_clbits’ are integers, which determine the amount of qubits or classical bits assigned to a ‘QuantumCircuit’ object. The attributes of the class are:

‘qubit_index_counter: ndarray’
An array used to keep track of the indices assigned to qubits within the circuit. It ensures that each qubit receives a unique and consistent index.

‘clbit_index_counter: ndarray’
Similar to the qubit counter, this array tracks the indices of classical bits in the circuit, ensuring correct mapping and ordering.

‘xla_mode: int’
An integer flag determining whether the circuit is operating in XLA-compatible mode. This can influence how operations are constructed or validated.

‘abstract_params: Set[sympy.Symbol]’
A set containing all symbolic parameters used within the circuit. These parameters typically come from SymPy and allow for parameterized gates or symbolic analysis.

‘qubits: List[Qubit]’
A list of all quantum bits that belong to the circuit. Each element is a Qubit object representing a physical or logical qubit.

‘clbits: List[Clbit]’
A list of classical bits associated with the circuit. These are used for measurement outcomes or classical control flow.

‘data: List[Instruction]’
A sequential list of all instructions applied to the circuit. Each instruction includes an operation (e.g., a gate or measurement) and its target qubits and/or clbits.

Class methods:

In the qrisp documentation for ‘QuantumCircuit’ methods are divided into:

  • Constructing ‘QuantumCircuit’s
  • Evaluating ‘QuantumCircuit’s
  • Interfacing
  • Operation application methods
Constructing ‘QuantumCircuit’s:

  • ‘add_qubit’ — add a new quantum bit (Qubit) to the circuit.
  • ‘add_clbit’ — add a new classical bit (Clbit) to the circuit.
  • ‘copy’ — generate a deep copy of the circuit (i.e. duplicate the circuit, including its qubits, classical bits and instructions).
  • ‘clearcopy’ — create a copy of the circuit’s qubits and classical bits, but with no instructions — i.e. an “empty” circuit with the same registers.
  • ‘inverse’ — returns the inverse (dagger) of the quantum circuit (i.e. the reversed circuit with inverted operations).
  • ‘extend’ — extend (concatenate) this circuit with another ‘QuantumCircuit’, optionally applying a translation dictionary (e.g. mapping bits) to integrate them.
  • ‘bind_parameters’— for circuits with symbolic (abstract) parameters: return a new circuit where specified symbolic parameters (e.g. Sympy symbols) are bound to concrete values.
  • ‘transpile’ — transpiles the ‘QuantumCircuit’ in the sense that there are no longer any synthesized gate objects.
Evaluating ‘QuantumCircuit’s:
  • 'run' — runs the circuit on a backend and returns measurement results.
  • 'statevector_array' — simulates the circuit and returns the statevector as a NumPy array.
  • 'get_unitary' — computes the unitary matrix of the circuit.
  • 'compare_unitary' — compares this circuit’s unitary to another circuit’s unitary.
  • 'depth' — returns the circuit depth.
  • 't_depth' — returns the T-depth of the circuit.
  • 'cnot_depth' — returns the CNOT-depth of the circuit.
  • 'num_qubits' — returns the number of qubits in the circuit.
  • 'count_ops' — returns a count of all gate types in the circuit.
  • 'to_latex' — produces a LaTeX representation of the circuit.
  • 'qasm' — exports the circuit in QASM format.
Interfacing:
  • 'from_qiskit' — creates a ‘QuantumCircuit’ from an existing Qiskit circuit.
  • 'from_qasm_str' — constructs a ‘QuantumCircuit’ from a QASM string.
  • 'from_qasm_file' — loads a ‘QuantumCircuit’ from a QASM file.
  • 'to_qiskit' — converts the circuit into a Qiskit ‘QuantumCircuit’.
  • ‘to_pennylane’ — method to convert the given ‘QuantumCircuit’ to a Pennylane Circuit.
  • ‘to_pytket’ — method to convert the given QuantumCircuit to a PyTket Circuit.
Operation application methods:
  • 'measure' — adds a measurement operation to one or more qubits, optionally storing results in classical bits.
  • 'x' — applies a Pauli-X (NOT) gate to the target qubit(s).
  • 'y' — applies a Pauli-Y gate to the target qubit(s).
  • 'z' — applies a Pauli-Z gate to the target qubit(s).
  • 'h' — applies a Hadamard gate, creating equal superpositions.
  • 'cx' — applies a controlled-X (CNOT) gate with one control and one target.
  • 'cy' — applies a controlled-Y gate.
  • 'cz' — applies a controlled-Z gate.
  • 'mcx' — applies a multi-controlled X gate with arbitrarily many control qubits.
  • 'ccx' — applies a Toffoli gate (double-controlled X).
  • 'rx' — applies a rotation around the X axis.
  • 'ry' — applies a rotation around the Y axis.
  • 'rz' — applies a rotation around the Z axis.
  • 'p' — applies a phase gate.
  • 'cp' — applies a controlled-phase gate.
  • 'u3' — applies a general single-qubit U3 rotation (general Euler-rotation gate).
  • 'swap' — swaps the states of two qubits.
  • 't' — applies a T-gate (pi/8 phase).
  • 't_dg' — applies the inverse of the T-gate.
  • 's' — applies an S-gate (sqrt(Z) phase).
  • 's_dg' — applies the inverse of the S-gate.
  • 'sx' — applies the square-root of X gate.
  • 'sx_dg' — applies the inverse of the SX gate.
  • 'rxx' — applies a two-qubit XX-rotation.
  • 'rzz' — applies a two-qubit ZZ-rotation.
  • 'xxyy' — applies a parameterized XXYY-rotation on two or more qubits.
  • 'reset' — resets qubits to the computational |0⟩ state.
  • 'unitary' — applies an arbitrary custom unitary matrix to specific qubits.
  • 'gphase' — applies a global phase to the entire state.
  • 'id' — applies an identity operation (no-op), useful for alignment or placeholders.

Details and design decisions:

The class contains many methods, some of them are static, which is underlined in the diagram. Class functionality:

QuantumBool

Class functionality:

The ‘QuantumBool’ class is described in the official Qrisp documentation [33]:

“QuantumBools are the quantum type, which represents boolean truth values. They are the return type of comparison operators like the equality ==.

Apart from their behavior as a QuantumVariable, they can also be treated like ControlEnvironments.”

Class constructor and attributes:

The class constructor optionally receives two parameters:

  • qs: QuantumSession - this ‘QuantumSession’ is optionally used to instantiate the parent class constructor
  • name: String – this string is optionally used to instantiate the parent class constructor

The class attribute are:

  • env: ClControlEnvironment | ControlEnvironment – this attribute is instantiated in the __enter__ method. ‘QuantumBool’ objects can be used as environments.
  • qfloat_comparison: bool - this bool is instantiated as false. It is unclear when this attribute is used.
  • size: int – this attribute is actually a method with the property decorator, making it an immutable size attribute, constantly being 1

Class methods:

Many of the methods of this class are dunder methods. In the following the dunder methods will be listed separated from the non-dunder methods.

The dunder methods are as follows:

  • ‘__and__’ – Implements quantum logical AND using a multi-controlled X (mcx) gate and returns a new ‘QuantumBool’
  • ‘__or__’ – Implements quantum logical OR via De Morgan-style construction using mcx and x gates, returning a new ‘QuantumBool’
  • ‘__xor__’ – Implements quantum logical XOR using controlled-X (cx) gates and returns a new ‘QuantumBool’
  • ‘__invert__’ – Implements quantum logical NOT by copying the qubit via cx and flipping the result, returning a new inverted ‘QuantumBool’
  • ‘__enter__’ – This method is called by the ‘with’ statement and enters a quantum control environment controlled by the first qubit
  • ‘__exit__’ – Exits the control environment and automatically uncomputes the ‘QuantumBool’
  • ‘__bool__’ – This prevents conversion to classical bools
  • ‘__len__’ – This method returns the length of the ‘QuantumBool’ which is always 1

The non-dunder methods are as follows:

  • ‘encoder’ – Converts a classical bool or integer value into its encoded integer representation for the quantum register
  • ‘decoder’ – Converts a measured integer value back into a classical bool (or a JAX bool array in tracing mode)
  • ‘flip’ – This method flips the truth value of a ‘QuantumBool’ object
  • ‘jdecoder’ – Returns a JAX bool array representation of a decoded measurement result

Details and design decisions:

This class has multiple dunder methods and can be used as an environment. The dunder methods are labelled with ‘<<>>’ notation. Furthermore the class has a ‘ControlEnvironment’ or ‘ClControlEnvironment’ composition (the ‘env’ attribute can be either). The class inherits from the ‘QuantumVariable’ class.

QuantumColor

Class functionality:

The ‘QuantumBool’ class is described in the code of its implementation [34]:

“The QuantumColor is a custom QuantumVariable implemented with tackling the Max-k-Colorable-Subgraph problem and other coloring optimization problems in mind. It provides flexibility in choosing encoding methods and leverages efficient data structures like QuantumArrays to enhance computational performance.

 The QuantumColor class takes as input a list of colors and a flag indicating the preferred encoding method - binary or one-hot encoding. The choice of encoding method has implications for how colors are represented in the quantum computation.

In binary encoding, each color is represented by a unique binary number. For instance, if there are four colors, Red, Green, Blue, and Yellow, they can be represented, for example, as [0,0], [0,1], [1,0], and [1,1] respectively.

In contrast, one-hot encoding represents each color as an array where only one element is 1 and the rest are 0. Using the same four-color example, red can be represented as [1,0,0,0], green as [0,1,0,0], blue as [0,0,1,0], and yellow as [0,0,0,1].

Another key feature of the QuantumColor class is its use of QuantumArrays. QuantumArrays are data structures designed for efficient quantum computation. They allow for compact representation and manipulation of quantum states and operators.”

Class constructor:

The class constructor receives two parameters, which get instantiated:

  • list_of_colors: List<Integer> - “The list of colors to be used in the coloring problem instance” [34]
  • one_hot_enc: bool – “The flag to indicate whether to use one-hot encoding. If False, binary encoding is used. Default is True.“ [34]

Class methods:

The class has one method ‘decoder’, which returns the decoded color, if existing.

Details and design decisions:

The class inherits from the ‘QuantumVariable’ class.

QuantumFloat

Class functionality:

As described in the official Qrisp documentation [35]:

“This subclass of QuantumVariable can represent floating point numbers (signed and unsigned) up to an arbitrary precision.

The technical details of the employed arithmetic can be found in this article.”

Class constructor and attributes:

The class receives 5 parameters:

  • ‘msize: int’ – The amount of qubits the ‘QuantumFloat’ holds. The size depends on the sign of the ‘QuantumFloat’. It becomes instantiated by a method with the ‘@attribute’ decorator, converting the method into an attribute.
  • exponent: int – This integer determines the precision and value range of the ‘QuantumFloat’. It is by default 0 and becomes instantiated.
  • qs: QuantumSession – This ‘QuantumSession’ is the session in which the ‘QuantumFloat’ gets embedded in. This parameter is used for the parent constructor.
  • name: String – “A name which uniquely identifies the QuantumVariable. If ended with a *, name is allowed to be updated if a naming collision arises. By default, Qrisp will try to infer the name of the Python variable - otherwise a generic name is given“ [35]. It is by default null and is used in the parent constructor.
  • Signed: bool – this bool determines if the ‘QuantumFloat’ is able to represent negative integers. It is by default false and becomes instantiated.

The class attributes are:

  • static_attributes: List<String> - This attribute defines object attributes that are treated as constant for JAX-Tracing.
  • mshape: (int, int) – This attribute describes the significance range of the mantissa in powers of 2

Class methods:

The class has many dunder methods that alter the functionality python uses arithmetic, representation and other operations on a ‘QuantumFloat’ object. In the following there is differentiation in dunder methods and standard methods:

Dunder methods:

  • ‘__add__’ – this dunder method changes the logic of the addition operation (+) for ‘QuantumFloat’. It enables quantum addition between two ‘QuantumFloat’ objects or between a ‘QuantumFloat’ and a classical number.
  • ‘__sub__’ – this dunder method changes the logic of the subtraction operation (-). It performs quantum subtraction between two ‘QuantumFloat’ objects or between a ‘QuantumFloat’ and a classical value.
  • ‘__mul__’ – this dunder method changes the logic of the multiplication operation (*). It allows multiplication between two ‘QuantumFloat’ objects or between a ‘QuantumFloat’ and an integer.
  • ‘__truediv__’ – this dunder method changes the logic of the division operation (/). It performs quantum division using the internal quantum division algorithm.
  • ‘__floordiv__’ – this dunder method changes the logic of the floor division operation (//). It performs integer division for unsigned ‘QuantumFloat’ values.
  • ‘__pow__’ – this dunder method changes the logic of the power operation ()**. It enables exponentiation of a ‘QuantumFloat’, including inversion when the exponent is -1.
  • ‘__radd__’ – this dunder method implements right-side addition so that addition works when a ‘QuantumFloat’ appears on the right side of the + operator.
  • ‘__rmul__’ – this dunder method implements right-side multiplication so multiplication works when a ‘QuantumFloat’ appears on the right side of *.

  • ‘__rsub__’ – this dunder method implements right-side subtraction, allowing expressions where the ‘QuantumFloat’ appears on the right side of -.

  • ‘__iadd__’ – this dunder method changes the logic of the in-place addition operation (+=). It modifies the current ‘QuantumFloat’ directly instead of creating a new one.

  • ‘__isub__’ – this dunder method changes the logic of the in-place subtraction operation (-=). It subtracts a value directly from the existing ‘QuantumFloat’.
  • ‘__imul__’ – this dunder method changes the logic of the in-place multiplication operation (*=). It multiplies the ‘QuantumFloat’ by a classical value directly in the same register.
  • ‘__ilshift__’ – this dunder method changes the logic of the left bit shift operation (<<=). It increases the exponent of the ‘QuantumFloat’.
  • ‘__irshift__’ – this dunder method changes the logic of the right bit shift operation (>>=). It decreases the exponent of the ‘QuantumFloat’.
  • ‘__lt__’ – this dunder method changes the logic of the less-than comparison (<) between ‘QuantumFloat’ values or classical numbers and returns a ‘QuantumBool’.
  • ‘__gt__’ – this dunder method changes the logic of the greater-than comparison (>) and returns a ‘QuantumBool’.
  • ‘__le__’ – this dunder method changes the logic of the less-than-or-equal comparison (<=).
  • ‘__ge__’ – this dunder method changes the logic of the greater-than-or-equal comparison (>=).
  • ‘__eq__’ – this dunder method changes the logic of the equality comparison (==) between ‘QuantumFloat’ objects or classical values and returns a ‘QuantumBool’.
  • ‘__ne__’ – this dunder method changes the logic of the inequality comparison (!=).
  • ‘__hash__’ – this dunder method defines the hash value of the object, allowing ‘QuantumFloat’ instances to be used in hashed data structures such as dictionaries or sets.

Normal methods:

  • ‘decoder’ – this method converts a basis state index into the classical number represented by the ‘QuantumFloat’. For signed numbers it applies the signed integer isomorphism and then scales the result by the exponent.
  • ‘jdecoder’ – this method provides a JAX-compatible version of the decoder, returning the decoded value during tracing or JIT execution.
  • ‘encoder’ – this method converts a classical numeric value into the integer basis state index used internally by the ‘QuantumFloat’ representation.
  • ‘sb_poly’ – Described in the code [34]: ‘Returns the semi-boolean polynomial of this ‘QuantumFloat’ where ‘m’ specifies the image extension parameter.’
  • ‘encode’ – this method initializes the ‘QuantumFloat’ with a given classical value. If rounding is enabled, the value is first rounded to the closest representable number before encoding.
  • ‘exp_shift’ – described in the code [34]: ‘Performs an internal bit shift. Note that this method doesn't cost any quantum gates. […]‘
  • ‘add_sign’ – described in the code [34]: ‘Turns an unsigned QuantumFloat into its signed version.’
  • ‘sign’ – described in the code [34]: ‘Returns the sign qubit. This qubit is in state |1> if the QuantumFloat holds a negative value and in state |0> otherwise.’
  • ’init_from’ – this method initializes the ‘QuantumFloat’ with the value stored in another ‘QuantumFloat’, copying the encoded state between registers while checking for overflow or rounding issues.
  • ‘incr’ – this method increments the ‘QuantumFloat’ object depending on the exponent or a given value x using a quantum increment operation.
  • ‘significant’ – described in the code [34]: ‘Returns the qubit with significance k’
  • ‘truncate’ – described in the code [34]: ‘Receives a regular float and returns the float that is closest to the input but can still be encoded.’
  • ‘get_ev’ – described in the code [34]: ‘Retrieves the expectation value of self.
  • ‘quantum_bit_shift’ – described in the code [34]: ‘Performs a bit shift in the quantum device. While ‘exp_shift’ performs a bit shift in the compiler (thus costing no quantum gates) this method performs the bitshift on the hardware. This has the advantage, that it can be controlled if called within a ‘ControlEnvironment’ and furthermore admits bit shifts based on the state of a ‘QuantumFloat’.

Details and design decisions:

This class has many dunder methods, which also have the ’@gate_wrap’ annotation. For these methods the <<>> notation holds multiple terms to describe the behavior. The class has two methods ‘msize’ and ‘mshape’ that are declared with the ‘@property’ declaration. These attributes are shown in the diagram as normal attributes, though they are methods that implement a special logic for the attributes. ‘QuantumFloat’ inherits from the parent class ‘QuantumVariable’.

OHQInt

Class functionality:

‘OHQInt’ is a specialized quantum integer type used internally for optimized high-level quantum arithmetic

Class methods:

The class has the following methods:

  • ‘decoder’ – this method takes an integer i and checks if it is to the power of two. If yes it will return the log of base 2 of the integer as a integer, otherwise it will return the integer -3.
  • ‘__eq__’ – this dunder method changes the == operation for ‘OHQInt’ objects and returns a ‘QuantumBool’ that is controlled via CNOT with the specified Qubit ‘other’. (in case other is not a int, the method raises an exception)
  • ‘__hash__’ – this method is a copy of the ‘hash’ of the parent class ‘QuantumVariable’
  • ‘is_even’ – this method returns a ‘QuantumBool’ that is controlled via CNOT and the even Qubits from the ‘OHQInt’ object.
  • ‘is_odd’ – this method returns a ‘QuantumBool’ that is controlled via CNOT and the odd Qubits from the ‘OHQInt’ object.
  • ‘__lt__’ – this method returns a ‘QuantumBool’ that is controlled via CNOT and the Qubits of the ‘OHQInt’ that have a smaller order than the specified ‘other’ integer. (If ‘other’ is not a integer the method raises a exception)

Details and design decisions:

This class does not have unique attributes or a constructor. It has dunder methods that alter the behavior of representations and operations on the object. They are noted with the <<>> notation. The class inherits from the parent class ‘QuantumVariable’.

QuantumArray

Class functionality:

As described in the official Qrisp documentation [36]:

“This class allows the convenient management of multiple ‘QuantumVariable’ objects of one type. Inspired by the well known numpy ndarray, the ‘QuantumArray’ supports many convenient array manipulation methods.”

Class constructor and attributes:

The ‘QuantumArray’ class constructor has 3 parameters:

  • ‘qtype: QuantumVariable’ – this parameter is central to this class and becomes instantiated. It needs to be defined inside a tracing context. It is defined as an attribute with the ‘@property’ annotation and can be a List of ‘QuantumVariable’ objects too, if outside of a tracing context.
  • ‘shape: int | np.integer’ – this attribute can be a normal integer or a numpy integer. It determines the shape of the ‘QuantumArray’ object. It is defined as an attribute with the ‘@property’ annotation and is a tuple of integers.
  • ‘qs: QuantumSession’ – this is the ‘QuantumSession’ object in which the ‘QuantumArray’ object becomes embedded in. ‘qs’ gets instantiated within the constructor and is optional.

The attributes of the class are:

  • ‘ind_array: jnp.array’ – a jax numpy array used as a central data structure to enable fast compilation to the ‘QuantumArray’.
  • ‘qtype_template: QuantumVariableTemplate’ – a template used to construct the ‘qtype’ with a ‘DynamicQubitArray’
  • ‘qs: QuantumSession’ – this ‘QuantumSession’ object is important for handling session data, such as tracing qubits.
  • ‘qb_array: DynamicQubitArray’ – this ‘DynamicQubitArray’ is used to construct the ‘qtype’ and is used for tracing
  • ‘qv_list: List<QuantumVariable>’ – this List of ‘QuantumVariable’ objects is used for indexing the ‘QuantumVariables’ the ‘QuantumArray’ holds.
  • ‘size: int’ – This is the size of the ‘QuantumArray’ object.
  • ‘ndim: int’ – This is how many dimensions the ‘ind_array’ object has.

Class methods:

The class has many dunder methods that alter the functionality how Qrisp uses arithmetics, representation and other operations on a ‘QuantumArray’ object. Furthermore, this class contains internal helper methods that support the implementation of arithmetic operations. In the following is a differentiation in dunder methods, standard methods and internal helper methods:

Dunder methods:

  • ‘__getitem__’ – this dunder method changes the logic of the indexing operation. It allows retrieving either a single ‘QuantumVariable’ or a sliced ‘QuantumArray’. The method also supports quantum indexing, where a ‘QuantumFloat’ can be used as an index, returning a ‘QuantumEnvironment’ containing the selected array element.
  • ‘__setitem__’ – this dunder method changes the logic of assignment to array elements. It encodes classical values into the selected slice or element of the ‘QuantumArray’.
  • ‘__matmul__’ – this dunder method changes the logic of the matrix multiplication operator (@). It performs quantum matrix multiplication between two ‘QuantumArray’ objects or between a ‘QuantumArray’ and a classical NumPy matrix.
  • ‘__rmatmul__’ – this dunder method implements right-side matrix multiplication so that multiplication works when a ‘QuantumArray’ appears on the right side of the @ operator.
  • ‘__iter__’ – this dunder method enables iteration over the elements of the ‘QuantumArray’, returning a ‘QuantumArrayIterator’ that iterates through the flattened array elements.
  • ‘__array_ufunc__’ – this dunder method integrates the ‘QuantumArray’ with NumPy’s universal function system. It intercepts operations such as np.matmul and redirects them to the corresponding ‘QuantumArray’ implementation.
  • ‘__len__’ – this dunder method defines the length of the ‘QuantumArray’. It returns the length of the first dimension of the internal index array.
  • ‘__str__’ – this dunder method defines the string representation of the ‘QuantumArray’. In normal mode it returns the measurement results of the array, while in tracing mode it returns a placeholder string describing the array shape.
  • ‘_add_’ – this dunder method changes the logic of the addition operation. It performs element-wise quantum addition between two ‘QuantumArray’ objects or between a ‘QuantumArray’ and a classical scalar or array.
  • ‘__sub__’ – this dunder method changes the logic of the subtraction operation. It performs element-wise quantum subtraction between two arrays or between a ‘QuantumArray’ and classical values.
  • ‘__mul__’ – this dunder method changes the logic of the multiplication operation. It performs element-wise quantum multiplication between arrays or between a ‘QuantumArray’ and classical values.
  • ‘__eq__’ – this dunder method changes the logic of the equality comparison. It performs element-wise comparison between arrays or classical values and returns a ‘QuantumArray’ of ‘QuantumBool’.
  • ‘__ne__’ – this dunder method changes the logic of the inequality comparison. It returns a ‘QuantumArray’ containing the result of element-wise inequality tests.
  • ‘__gt__’ – this dunder method changes the logic of the greater-than comparison. It returns a ‘QuantumArray’ of ‘QuantumBool’ values.
  • ‘__ge__’ – this dunder method changes the logic of the greater-than-or-equal comparison.
  • ‘__lt__’ – this dunder method changes the logic of the less-than comparison.
  • ‘__le__’ – this dunder method changes the logic of the less-than-or-equal comparison.
  • ‘__iadd__’ – this dunder method changes the logic of the in-place addition operation. It performs element-wise quantum addition and updates the array directly.
  • ‘__isub__’ – this dunder method changes the logic of the in-place subtraction operation. It subtracts values from the existing ‘QuantumArray’.
  • ‘__imul__‘– this dunder method changes the logic of the in-place multiplication operation. It multiplies the current ‘QuantumArray’ by another array or classical value in place.
  • ‘__lshift__’ – this dunder method changes the logic of the left shift operator (<<). It allows injecting a quantum function into the array so the function operates on each element.
  • ‘__lshift_o__’ – this method is a variant of the injection operator that redirects quantum functions to operate on the entire ‘QuantumArray’.

Normal methods:

  • ‘encode’ – this method initializes the ‘QuantumArray’ with classical values by encoding them into the quantum registers. It sets the array into the corresponding computational basis state.
  • ‘reshape’ – this method changes the shape of the ‘QuantumArray’ without allocating new qubits. It returns a view with the specified shape, similar to NumPy’s reshape.
  • ‘flatten’ – this method flattens the ‘QuantumArray’ into a one-dimensional array view without allocating new qubits.
  • ‘ravel’ – this method ravels the array into a one-dimensional view, similar to NumPy’s ravel.
  • ‘transpose’ – this method rearranges the axes of the ‘QuantumArray’, returning a view with the axes permuted. The method can receive multiple integers (thus n*int)
  • ‘swapaxes’ – this method swaps two axes of the ‘QuantumArray’, returning a new view of the array.
  • ‘concatenate’ – this method concatenates two ‘QuantumArray’ objects along a specified axis. The arrays must have compatible ‘qtype’ definitions.
  • ‘delete’ – this method deletes all ‘QuantumVariables’ contained in the array by releasing their qubits. Optionally, it can verify that the qubits are in the |0⟩ state.
  • ‘duplicate’ – this method creates a fresh ‘QuantumArray’ with the same ‘qtype’ and shape. If the ‘init’ parameter is set to True, the values from the original array are copied into the new one.
  • ‘measure’ – this method measures each element of the ‘QuantumArray’ and returns a classical array containing the measurement results.
  • ‘get_measurement’ – this method executes the ‘QuantumCircuit’ and retrieves the measurement distribution of the entire array. The results are returned as a dictionary mapping ‘OutcomeArray’ objects to probabilities.
  • ‘decoder’ – this method converts a measurement result (represented as an integer basis state index) into a classical ‘OutcomeArray’ representing the measured values of the ‘QuantumArray’.
  • ‘most_likely’ – this method performs a measurement and returns the most probable outcome state of the array.

Internal helper methods:

  • ‘_element_wise_out_of_place_injection’ – this internal method applies a function element-wise and returns a new ‘QuantumArray’ containing the results.
  • ‘_element_wise_in_place_call’ – this internal method applies a function element-wise directly on the existing ‘QuantumArray’.
  • ‘_validate_arithmetic’ – this internal method validates the compatibility of operands for element-wise arithmetic operations, checking both type and shape.

The class has 2 aggregations: ‘QuantumSession’ and a List of ‘QuantumVariable’ objects, as they can exist outside a ‘QuantumArray’ object.
